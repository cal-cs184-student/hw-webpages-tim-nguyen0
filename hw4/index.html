Overview

In this write-up I document a complete cloth simulation pipeline based on a mass-spring model. Starting from a rectangular cloth parameterization, I build a row-major grid of point masses, connect them with structural, shearing, and bending springs, and simulate motion using force accumulation and Verlet integration, with additional deformation constraints to keep the system stable. I then extend the system with collisions against scene primitives (sphere and plane), cloth self-collision via spatial hashing, and a set of GLSL shaders for real-time lighting, texturing, bump and displacement mapping, and mirror-like environment reflections.

Throughout, we relate each pipeline stage to its visible effect: physically, parameters such as spring stiffness, damping, and density noticeably change how the cloth falls and settles; visually, shader choices strongly change perceived material quality even when geometry is unchanged. My main engineering challenge was balancing realism and stability; in particular, self-collision and spring-length correction required tuning to avoid things like excessive jitter or clipping or other unstable behavior.

Part 1: Masses and Springs

Implementation

This part works in two stages, similar to how I usually describe rasterization steps. First, I create every point mass in row-major order. For a horizontal sheet, points are placed in xz with y fixed at 1. For a vertical sheet, points are placed in xy with a small random z offset so the cloth does not start perfectly flat. I map 2D indices to the linear array with idx = y * num_width_points + x.

Second, I connect springs after all points exist. I add structural springs to left and upper neighbors, shearing springs to upper-left and upper-right, and bending springs two points away in the same row or column. I also check pinned coordinates while building the grid, so fixed points are locked from the start. This order avoids boundary bugs and keeps the mesh setup predictable.

Part 2: Numerical Integration

Implementation

This simulation step also breaks into stages. First, I clear and accumulate forces: gravity/external accelerations, then spring forces with equal and opposite updates at each endpoint. If a spring type is disabled in the GUI, I skip it. Bending uses a smaller scale factor so it behaves softer than structural and shearing springs.

Next, I update positions with damped Verlet, then run collisions, then apply the 10% spring-length clamp. In practice, this order made behavior much more stable for me than trying to clamp too early. The cloth keeps its shape better and I get fewer exploding frames when parameters are pushed.

Parameter Study

Spring constant ks: low ks gives a stretchier sheet with deeper sag and larger folds; high ks looks stiffer and holds shape. I kept everything else fixed while changing only ks so the comparison is clean.

Density: higher density makes the sheet feel heavier and hit objects harder. Lower density makes motion lighter and slower.

Damping: higher damping kills motion faster, while lower damping keeps oscillations around longer.

Part 3: Collisions with Primitives

Implementation

I handled primitive collisions in two straightforward checks. For spheres, I check if a point is inside the radius and, if so, project it back to the surface along the radial direction, with friction scaling applied from the previous position.

For planes, I check whether the point crossed the plane between the last and current position. If it did, I compute the intersection point, offset slightly to stay on the correct side, then apply friction in the same style as the sphere case. Every point mass is tested against each collision object each step.

The screenshots in this section show baseline primitive collisions using one sphere scene and one plane scene. The sphere case highlights smooth curved contact and draping around convex geometry, while the plane case shows broad contact and settling on a flat support.

Part 4: Self-Collisions

Implementation

I handle self-collision with spatial hashing to avoid an O(n^2) all-pairs check. In hash_position, I partition space into 3D boxes with dimensions w = 3*width/num_width_points, h = 3*height/num_height_points, and t = max(w, h), and hash integer cell coordinates into a scalar key. In build_spatial_map, I clear the previous map and insert each point-mass pointer into its bucket.

In self_collide, each point mass considers only candidates in the same bucket. For neighbors within 2 * thickness, I compute a repulsion correction that pushes the mass just outside the threshold. I average pairwise corrections and scale by simulation_steps before application, skipping self-pairs and pinned masses. This design keeps runtime practical while significantly reducing cloth self-intersection.

Part 5: Shaders

Conceptual Explanation

Following common usage, we treat a shader program as a matched pair of GPU stages (vertex + fragment) that run massively in parallel during rendering. The vertex shader transforms per-vertex attributes (position, normal, tangent, uv) into world/screen-space values and passes interpolated varyings to the fragment shader. The fragment shader then computes the final per-fragment color from those varyings plus scene uniforms (lights, textures, camera position, material parameters). In the pipeline I built, this separation makes it practical to combine geometric transformations with material and lighting effects in real time.

Shader Modes Used

In this implementation, Phong is available as a single shader mode rather than separate runtime toggles for ambient-only, diffuse-only, and specular-only outputs. I therefore present one representative Phong result and then compare it with the other available shaders in the dropdown: Normal, Diffuse, Texture, Bump, Displacement, and Mirror.

Beyond Blinn-Phong, I also provide diffuse shading, texture mapping, bump mapping, displacement mapping, and environment-mapped mirror reflection. For bump versus displacement: bump mapping perturbs normals only (lighting detail without changing underlying geometry), while displacement mapping moves vertex positions and therefore changes silhouette and geometry, especially on coarse meshes.

Extra Credit

Spatially varying wind

I add an external force per point mass from a smooth vector field built from sin and cos of position and time, with strength and frequency set in the cloth JSON (enable_wind, wind_strength, wind_frequency). Compared with gravity alone, the sheet folds and drifts along a time-varying direction instead of staying in one sagging plane. The animation below uses scene/extra_wind.json.

Self-collision: AABB BVH vs. uniform spatial hashing

The default path hashes each particle into a cell and tests only masses stored in that bucket. I also provide an optional axis-aligned BVH over the particles and query the tree with a neighborhood of radius about 2 * thickness around each point. The self-collision correction is the same as in the hash path; only candidate selection changes.

OpenMP optimization for spatial hash self-collision

For the spatial hash path I parallelized self-collision with OpenMP using a snapshot-and-apply scheme: copy positions, accumulate per-particle correction vectors in parallel, then apply them in a separate pass so threads do not write the same positions concurrently. The build enables this when OpenMP is found; otherwise the code uses the original sequential loop.

Pressing B while the app runs records several hundred steps and prints average step time and self-collision time, together with omp_build and parallel_self_coll so runs are comparable across configurations (BVH self-collision does not use the OpenMP hash loop, so that flag reads n/a there).

Moving objects and cloth from the GUI

I added a Scene (extra) NanoGUI window with numeric fields for the sphere center, a plane reference point, and rigid translations of the box and capsule. A separate “Cloth rigid shift” applies a one-time translation to all point masses (current, previous, and rest positions) so the cloth can be repositioned without editing scene files. Together with pause and step, this makes it straightforward to adjust obstacles or the sheet and compare outcomes under fixed parameters.

Additional rigid primitives

Beyond the sphere and plane required in the assignment, I support collision and scene loading for an axis-aligned box and a capsule (segment between a and b), using the same friction-weighted projection style as the sphere. The box pushes interior particles to the nearest face along the shortest penetration axis; the capsule uses the closest point on the segment. These primitives produce edge and elongated contacts that differ from sphere-only setups. Example scenes: scene/extra_box.json, scene/extra_capsule.json.
